== Abstract

The OGC API – Environmental Data Retrieval (EDR) Part 1: Core Standard was designed to be flexible and easy to understand and implement by Web developers. As OGC API-EDR has been widely adopted, communities of practice have identified the need to constrain certain aspects of this flexibility in order to improve interoperability between implementations within specific domains. These constraints are expressed through API-EDR Service Profiles, which define additional, domain‑specific requirements on top of the Core Standard.

The OGC API-EDR Part 3: Service Profile Standard defines a consistent and interoperable approach for specifying such profiles. Its objective is to ensure that independent API-EDR implementations expose predictable and compatible behaviour when claiming conformance to the same service profile.

An OGC API-EDR Service Profile defines a constrained and well‑specified set of requirements that an API-EDR service shall implement to support a clearly defined set of functionality. In particular, service profiles establish standardized rules for the definition and use of collections and when appropriate their instances, ensuring that services claiming conformance present a consistent interface to clients. Conformance to a service profile indicates that an implementation meets all requirements defined by that profile.

An OGC API-EDR implementation may support multiple service profiles. Each supported profile represents a distinct conformance target and imposes specific requirements on the service’s behaviour and capabilities. Implementations shall advertise the service profiles they support through their conformance declaration, enabling clients to determine which profiles are available.

Each OGC API-EDR Service Profile consists of an OGC ModSpec compliant document that defines the service profile’s compliance rules. These rules are classified as requirements (which are subject to testing) and recommendations or permissions (which are not tested).
For every requirement defined in the document, a corresponding conformance test is provided to validate compliance with the service profile.

If a service profile introduces or restricts functionality that can be expressed using OpenAPI, it must also include an OpenAPI document that describes this functionality. A service profile OpenAPI definition, provides a precise, machine‑readable description of the operations, parameters, and responses required by the profile. These OpenAPI definitions are intended solely to describe the functionality defined by the service profile and do not represent a complete service description or replace the primary OGC API-EDR OpenAPI definition.

Service profile OpenAPI definitions are distinct from the OpenAPI definition of an OGC API-EDR implementation. The implementation OpenAPI definition represents the full set of capabilities offered by the service, including functionality defined by one or more supported service profiles as well as any additional, non‑profiled functionality. This separation allows service profiles to precisely define interoperable subsets of behaviour while enabling implementations to evolve and extend beyond the scope of individual profiles without compromising conformance.

== Preface

[NOTE]
====
The aim of the OGC API-EDR Part 3: Service Profile standard is to ensure interoperability between EDR API implementations by defining a consistent approach to specifying a Profile of OGC API-EDR Part 1: Core. To achieve this, it is essential that service providers use a consistent approach when defining `collections` and `instances` of `collections`. An OGC API-EDR Profile will specify a set of Requirements that an EDR API implementation must support to be a profile-compliant implementation.

This standard specifies Requirements and Recommendations for a Profile definition, and also conforms to the OGC Mod Spec Standard.

It is envisaged that this approach may be useful for other OGC API Standards.
