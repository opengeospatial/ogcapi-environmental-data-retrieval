I.  Abstract

The OGC API – Environmental Data Retrieval (EDR) Part 1: Core Standard offers a flexible framework for geospatial data access. However, this flexibility can lead to implementation variability, making it difficult for clients to interact with different services consistently. To address this, communities of practice use API-EDR Service Profiles to formalize domain-specific constraints that ensure predictable service behavior and simplify client integration.

The OGC API-EDR Part 3: Service Profile Standard establishes a consistent approach for specifying these profiles. Each profile serves as a normative “contract” for a specific domain, consisting of an OGC ModSpec-compliant document containing:

  • Requirements: Mandatory rules subject to formal testing and conformance validation.

  • Recommendations and Permissions: Guidelines and best practices that are not subject to testing.

The primary purpose of a Service Profile is to eliminate the “validation gap” by providing a machine-readable set of rules for service discovery and interaction. By standardizing collection structures and parameters, profiles allow clients to understand exactly what a specific implementation provides without manual intervention. To support this, each profile includes a dedicated OpenAPI definition that provides a precise, machine-readable description of the operations and responses required by that profile.

Service profiles utilize a tiered architecture to maintain a clear distinction between standard rules and deployed instances:

  • Core OpenAPI Definition: The base framework provided by Part 1.

  • Service Profile OpenAPI Definition: A specialized subset describing only the requirements of the profile.

  • Implementation OpenAPI Definition: The full description of a deployed service, which may incorporate multiple profiles alongside vendor-specific capabilities.

A Service Profile is not required to constrain all aspects of the Core Standard; profiles may selectively address only those elements relevant to their specific domain requirements.

Implementations shall advertise their supported service profiles through a conformance declaration. This enables clients to automatically discover available functionality while allowing services to evolve and extend beyond the scope of individual profiles without compromising conformance.

1.  Scope

This Standard specifies the methodology for defining Service Profiles for the OGC API – Environmental Data Retrieval (EDR) Part 1: Core. It is limited to the definition of restrictive profiles, which specialize the existing Core Standard to meet domain-specific needs.

Service Profiles may selectively constrain specific aspects of the Core Standard based on domain needs. A profile is not required to address all possible constraints defined in this Standard.

This Standard does not define profiles that extend the EDR API with new functionality that falls outside the Core Standard’s specification. While a service profile may impose requirements additional to those in the Core, all such requirements must remain compliant with the underlying OGC API — EDR Part 1: Core Standard.

The normative restrictions in a Service Profile are defined as Requirements, which utilize JSON Schema fragments to enable formal, machine-readable conformance testing. While developed for EDR, the profiling mechanisms and requirements defined herein are intended to be applicable to other OGC API Standards and implementations.

6.  Context

6.1.  Standardization Goal

The goal of this Standard is to ensure interoperability between implementations of the OGC API — Environmental Data Retrieval Standard (OGC API-EDR).

The OGC API-EDR Standard does not try to address every possible application domain. Rather, it provides a foundation which can be tailored for a specific domain. The result of this tailoring is a domain specific “profile” of the EDR API Standard.

A significant risk to this approach is that, in the act of profiling, interoperability will be compromised. This risk can be mitigated by establishing rules for how the OGC API-EDR Standard can be profiled. The goal of this Standard is to define a set of rules sufficient to ensure interoperability while retaining the adaptability needed to support domain-specific requirements.

6.2.  Standardization Target Type

The Standardization Target Type for this Standard is the Service Profile definition. This Standard establishes the requirements for creating, documenting, and validating these profiles.

It is important to understand the following distinctions:

  • A Standard for Standards: This document provides the rules for writing profile specifications; it does not define the requirements for the EDR API software implementation itself.

  • ModSpec Alignment: This Standard is a profile of OGC ModSpec Model — Part 1: Core — A Standard for Designing and Writing Modular Standards (ModSpec), ensuring that all resulting Service Profiles follow a consistent modular structure.

  • Implementation: An “implementation” of this Standard is a completed Service Profile document (e.g., a MetOcean EDR Profile).

  • Foundational Model: The profiling methodology is based on ISO 19106:2004 Geographic information — Profiles.

The EDR API Items query is like an OGC API — Features Part 1: Core endpoint. The Items query offers some basic compatibility with API-Features for convenience. It may be used to catalog pre-existing EDR sampling features. The pre-existence of an EDR sampling feature may be because a particular query has been cached for later use, such as a monitoring location. Or there may be a catalog of spatio-temporal sampling features such as domains of anomalies in a dataset. A GeoJSON-compatible JSON-Schema is specified to document an EDR API query endpoint and valid query parameters including time range, parameters, and spatial characteristics. A service can define a custom GeoJSON schema in the OpenAPI definition for the service, with the default being the edr-geojson schema if no alternative is documented.
